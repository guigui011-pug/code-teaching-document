# Code Teaching Document Skill

## 核心原则

这个 skill 的目标是：**让代码能力弱的用户，能像读教材一样看懂你写的每一段代码，并且在代码修改后文档始终保持同步。**

把用户当作你的学生，而不是同事。

---

## 文档体系结构

每个项目需要维护以下 6 个文档文件，保存在 `docs/` 目录下：

```
docs/
├── 代码结构说明.md       ← 整体架构，每个文件/模块是干什么的
├── 模块调用关系.md       ← 模块之间如何互相调用，数据流向
├── 实验流程说明.md       ← 整个实验/项目从头到尾怎么跑
├── 修改记录.md           ← 每次改动了什么、为什么改
├── 问题与解决.md         ← 遇到过的坑、报错、以及怎么解决的
└── 论文写作素材.md       ← 适合写进论文的方法描述、实验细节
```

---

## 文档分卷规则（重要）

### 为什么需要分卷？

文档过长会带来以下问题：
- AI（Claude / ChatGPT）每次读入上下文有长度限制，文档太长会被截断，导致后面的内容读不到
- 人读起来也会因为过长而感到疲劳、找不到重点
- 更新文档时容易因为文件太大而出现编辑错误

### 分卷触发阈值

**每个文档超过 800 行时，必须分卷。**

（800 行约等于一个中等长度的教程章节，是人和 AI 都能舒适处理的上限。）

### 分卷方式

**命名规则：** 在原文件名后加数字，从 `2` 开始。

| 原文件 | 分卷后 |
|--------|--------|
| `代码结构说明.md` | `代码结构说明.md` + `代码结构说明2.md` + `代码结构说明3.md` … |
| `修改记录.md` | `修改记录.md` + `修改记录2.md` … |
| 其余文档同理 | 依此类推 |

### 分卷时必须做的事

1. **在旧文件末尾**添加导航提示：
   ```
   ---
   📄 本文档已达到长度上限，后续内容见：代码结构说明2.md
   ```

2. **在新文件开头**添加来源说明：
   ```
   ---
   📄 本文档是「代码结构说明.md」的续篇（第2卷）
   前一卷：代码结构说明.md
   ---
   ```

3. **分卷要在完整的章节边界处切割**，不要把一个模块的说明切成两半。比如，如果正在写 `train.py` 的详解，要么整个放在旧文件里，要么整个移到新文件，不能一半在这里、一半在那里。

4. **告知用户**："`代码结构说明.md` 已超过 800 行，我创建了 `代码结构说明2.md` 继续记录，两个文件加起来是完整的说明。"

### 哪些文档最容易触发分卷？

- **修改记录.md**：项目越做越长，每次改动都会追加，最容易超限 → 建议按"阶段"分卷，比如`修改记录.md`记录第一阶段，`修改记录2.md`记录第二阶段
- **问题与解决.md**：遇到的坑越来越多也会变长
- **代码结构说明.md**：项目文件多时也会很长

## 各文档的写作规范

### 1. `代码结构说明.md`

**格式模板：**

```markdown
# 代码结构说明

## 项目概览
[用2-3句话说明这个项目是做什么的，就像给完全不懂的人解释]

## 文件目录

```
项目根目录/
├── main.py          ← 主入口，程序从这里开始运行
├── model.py         ← 定义了神经网络的结构
├── train.py         ← 负责训练模型
├── utils.py         ← 工具函数，被其他文件调用
└── config.py        ← 所有参数配置都在这里
```

## 每个文件详解

### `main.py` — 程序主入口
**它做什么：** [一句话]
**为什么需要它：** [解释存在的意义]
**关键代码段：**
```python
# 这里粘贴最核心的几行代码
# 每行都写注释解释在干什么
```
**类比理解：** [用生活中的例子打比方，帮助理解]

[其他文件依此类推]
```

### 2. `模块调用关系.md`

**格式模板：**

```markdown
# 模块调用关系

## 数据流向图（文字版）

```
用户运行 main.py
    ↓ 读取参数
config.py（参数配置）
    ↓ 传给
train.py（训练逻辑）
    ↓ 调用
model.py（模型定义）  +  utils.py（工具函数）
    ↓ 输出
结果文件 / 日志
```

## 逐步调用说明

### 第一步：程序启动
- 入口：`main.py` 的第 X 行
- 做了什么：[说明]
- 调用了谁：`config.py` 里的 `load_config()` 函数

### 第二步：[继续补充]
...

## 关键函数说明

| 函数名 | 在哪个文件 | 输入是什么 | 输出是什么 | 用来干嘛 |
|--------|-----------|-----------|-----------|---------|
| `train_one_epoch()` | train.py | 数据batch | 损失值 | 跑一轮训练 |
| ... | ... | ... | ... | ... |
```

### 3. `实验流程说明.md`

**格式模板：**

```markdown
# 实验流程说明

## 实验目标
[这个实验想证明什么 / 想解决什么问题]

## 运行步骤（手把手版）

### 第一步：准备环境
```bash
pip install ...  # 安装依赖
```
[解释为什么需要这些库]

### 第二步：准备数据
- 数据放在哪里：`data/` 文件夹
- 数据格式是什么：[说明]
- 如果没有数据，怎么办：[说明]

### 第三步：运行训练
```bash
python main.py --epochs 100 --lr 0.001
```
**参数解释：**
- `--epochs 100`：训练100轮，轮数越多模型学得越充分，但也越慢
- `--lr 0.001`：学习率，控制每次更新参数的幅度，太大容易跑飞，太小收敛慢

### 第四步：查看结果
[怎么看输出、怎么判断实验成功]

## 预期输出
[运行成功后应该看到什么]
```

## 执行规范：Claude 必须做到的事

### 写新代码时

1. 写完代码后，**立即**更新 `代码结构说明.md` 和 `模块调用关系.md`
2. 更新 `实验流程说明.md`（如果流程有变化）
3. 在 `修改记录.md` 里记录"新增了什么"
4. 告诉用户：「我已更新了以下文档：[列出文件名]，你可以打开查看。」

### 修改已有代码时

1. **先**在 `修改记录.md` 里记录这次改动
2. **然后**更新受影响的其他文档
3. **最后**用一段话向用户解释："这次修改了 XXX，原因是 YYY，对你理解代码的影响是 ZZZ"

### 出现报错/解决问题时

1. 立即在 `问题与解决.md` 里记录

## 初始化流程

当用户第一次触发此 skill 时（比如说"帮我把代码写到文档里"），执行：

1. 检查项目目录，列出所有代码文件
2. 创建 `docs/` 文件夹
3. 依次生成上述 6 个文档
4. 告知用户："我已经建立了完整的代码教学文档体系，位于 `docs/` 文件夹。你可以从《代码结构说明》开始阅读，再看《实验流程说明》，就能大致理解整个项目。"

---

## 文档同步检查清单

每次 Claude 写代码或改代码后，自检：

- [ ] 代码结构说明.md 是否反映了最新文件结构？
- [ ] 模块调用关系.md 是否更新了新的函数/调用链？
- [ ] 修改记录.md 是否记录了这次改动？
- [ ] 如果有报错并解决，问题与解决.md 是否已更新？
- [ ] 用户是否能根据文档，自己从头到尾串一遍整个逻辑？
- [ ] **任何文档是否超过 800 行？如果是，立即分卷并在文件间添加导航提示。**
